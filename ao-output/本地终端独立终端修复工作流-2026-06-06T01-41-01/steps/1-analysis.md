该 bug 的核心原因是：不同的终端标签/窗口共享了同一个 Shell 进程与伪终端 (PTY)。具体可能由以下一个或多个因素导致：

1. **进程模型错误**  
   每个终端视图只应启动一个独立的子进程（如 `bash` / `zsh`）。若代码中只创建了一次 `Process` 对象，并将同一进程的 stdio 分配到所有终端视图上，就会让所有标签共用同一个 Shell 会话。

2. **伪终端 (PTY) 分配不当**  
   SwiftTerm 的 `LocalProcessTerminalView` 需要为每个实例分配独立的 PTY master-slave 对。如果采用全局 PTY 或未在每次创建新标签时生成新的 PTY，则多个终端视图会读写同一主设备，导致输入输出互相覆盖。

3. **终端实例对象复用**  
   开发者可能将 `SwiftTermController` 或 `LocalProcessTerminalView` 实现为单例，或通过 IBOutlet 关联了所有标签页的视图到同一个控制器对象。此时用户切换标签看到的实际上是同一个终端视图（只是通过 `isHidden` 或 `tab` 切换显示），从未创建第二个进程。

4. **工作目录与环境变量共享**  
   虽然这通常是症状而非根因，但如果 Shell 进程是共享的，那么所有标签自然共享同一当前工作目录、环境变量和 shell 状态（如历史、别名）。反过来，若刻意在创建新标签时不做环境隔离，也会强化共享的假象。

5. **信号与生命周期管理缺陷**  
   没有为每个标签维护独立的进程生命周期：关闭一个标签可能没有终止其进程，导致进程被复用；或者按下 Ctrl+C 时信号被传递到共享的 Shell，影响所有标签。

**结论**：必须确保每个终端标签拥有独立的 `Process`、独立的 PTY 以及独立的 `LocalProcessTerminalView` 实例，且视图与进程之间保持一对一映射。任何单例化或共享 I/O 通道的设计都会导致会话共享。
